    71→    71→2. Workbook Architecture Overview
    72→    72→The Al-Nassim Master Catalog workbook is a multi-sheet Excel file containing 63 sheets organized into
    73→    73→three functional layers:
    74→    74→Layer 1: Core Data (1 sheet)
    75→    75→ Master Catalog — the primary product table with 2,546 rows × 52 columns. This is the only sheet
    76→    76→that should be directly edited by catalog managers.
    77→    77→Layer 2: Reference & Taxonomy (6 sheets)
    78→    78→ Taxonomy Reference — defines the valid Department → Category → Subcategory hierarchy (21
    79→    79→entries).
    80→    80→ Inferred Classifications — 208 products whose classification was inferred (not direct-matched) during
    81→    81→the V2 taxonomy project, with reviewer actions.
    82→    82→ Classification Summary — roll-up of the V2 classification coverage (100% achieved).
    83→    83→ Validation Report — taxonomy validation results.
    84→    84→ Translation Report — translation pipeline summary.
    85→    85→ Name Rewrite Report / Report V2 — summaries of the name-rewrite passes.
    86→    86→Layer 3: Audit & Correction Logs (56 sheets)
    87→    87→These sheets document the 8 rounds of translation quality assurance (V3 through V8, plus override
    88→    88→batches). Each audit round has a paired Audit sheet (findings) and Corrections sheet (applied fixes). Key
    89→    89→audit sheets include:
    90→    90→ Translation Audit V3 — the master audit log with 2,546 rows, one per product, showing action
    91→    91→(UNCHANGED / CORRECTED / FLAG_REVIEW / APPLIED) and reason.
    92→    92→ Translation Audit Summary V3 — roll-up stats and top recurring issues.
    93→    93→ Name Rewrite Log V2 — 935 rows documenting phrase-first translation rewrites.
    94→    94→ Audit Change Log — 768 individual changes from the V2 audit.
    95→    95→ Translation Quality Corrections — 472 corrections from the quality audit.
    96→    96→ Cross-Validation Discrepancies V3 — 517 cross-validation findings.
    97→    97→ Semantic Translation Corrections — 402 semantic fixes.

   161→   161→Design Rationale
   162→   162→ Ordering: Identity first, then classification, then descriptions, then specs, then commercial, then SEO,
   163→   163→then internal QA. This mirrors the order in which a catalog manager reads a product record.
   164→   164→ Bilingual pairing: Bilingual pairs (Name AR/EN, Color AR/EN, Material AR/EN, SEO Title AR/EN, etc.)
   165→   165→are always adjacent to facilitate side-by-side review.
   166→   166→ Spec grouping: All dimension columns (Length, Width, Height, Diameter) are grouped together,
   167→   167→followed by their shared Dimension Unit.
   168→   168→ QA isolation: Internal/QA columns (Validation Status, Confidence Score) are at the far right so they
   169→   169→don't interfere with customer-facing data views.

   378→   378→Source Row (Column A, #1)
   379→   379→Distinct values: 2,546 | Non-null: 2,546 | Null: 0

   381→   381→original source workbook from which this catalog was
   382→   382→migrated. Used for traceability during audits and data

   386→   386→4. Validation Rules Must be a positive integer. Range: 1–2548. Must be

   388→   388→5. Field Kind Auto-generated integer
   389→   389→6. Valid Options N/A — auto-generated
   390→   390→7. Dependencies None
   391→   391→8. Default Values Auto-assigned sequentially when a new row is created.
   392→   392→9. Duplicates No — must be unique. Acts as the primary internal row

   394→   394→10. Business Rules Never edited manually. Used by audit sheets
   395→   395→(Translation Audit V3, V4, V5, V6, V7, V8) to reference

   399→   399→Product ID (Column B, #2)
   400→   400→Distinct values: 2,546 | Non-null: 2,546 | Null: 0
   401→   401→1. Purpose The globally unique product identifier used by the
   402→   402→ERP/PIM system. This is the canonical business key for
   403→   403→--- Page 17 ---
   404→   404→the product and never changes once assigned.
   405→   405→2. Data Type String (numeric, typically 12 digits)
   406→   406→3. Mandatory Yes
   407→   407→4. Validation Rules Must be a non-empty string. In practice always
   408→   408→numeric, 12 digits. Must be unique.
   409→   409→5. Field Kind Free text (but should be numeric)
   410→   410→6. Valid Options N/A — assigned by ERP
   411→   411→7. Dependencies None
   412→   412→8. Default Values Assigned by the ERP system at product creation time.
   413→   413→9. Duplicates No — must be globally unique.
   414→   414→10. Business Rules This is the primary key for integration with ERP, e-
   415→   415→commerce, and warehouse systems. Never reused
   416→   416→even after a product is retired. Sample values:
   417→   417→102000000126, 102000000135.
   418→   418→Sample values: '102000000126' | '102000000135' | '102000000433'
   419→   419→SKU (Column C, #3)
   420→   420→Distinct values: 2,546 | Non-null: 2,546 | Null: 0
   421→   421→1. Purpose Stock Keeping Unit — the warehouse-facing identifier
   422→   422→used for inventory management, picking, and stock
   423→   423→movements. Often equals Product ID in this catalog
   424→   424→but is conceptually distinct (a SKU can differ if the
   425→   425→same product is stored in multiple warehouses).
   426→   426→2. Data Type String (numeric, 12 digits)
   427→   427→3. Mandatory Yes
   428→   428→4. Validation Rules Must be non-empty. In this catalog it mirrors Product
   429→   429→ID. Must be unique.
   430→   430→5. Field Kind Free text
